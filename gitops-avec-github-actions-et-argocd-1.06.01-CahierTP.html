      Introduction

      Important : les tps sont très scriptés afin que les participants soient autonomes et ne soient jamais bloqués en attendant de l’aide. Cela permet aussi de rejouer tranquillement les scenarios après la formation. Il est préférable d'éviter le copier/coller, car on retient beaucoup moins qu'en saisissant réellement. Prendre le temps de comprendre les instructions et pourquoi elles sont proposées. Le but n’est évidemment pas de vérifier que les TPs fonctionnent. Ne pas hésiter à poser des questions si quelque chose n’est pas clair. Éviter cependant de prendre trop d’avance sur les tps, au risque de poser des questions abordées plus tard dans le cours.

      Les scripts sont généralement de la forme suivante :

      Réaliser une action plus ou moins complexe

      détails des actions à effectuer pour réaliser l'action complexe

      ou informations pour expliquer l'action

      ou remarque particulière

      ou encore un bloc de code

      commandes à saisir
Sortie attendue des commandes (en gras les éléments notables)

      Il est bien sûr possible d'expérimenter, mais il vaut mieux éviter de trop s'écarter du scénario pour ne pas se retrouver dans une situation complexe. Idéalement, expérimenter dans un autre dépôt.

      TP 1 : Initialisation de l'usine logicielle (Le Fork)

      L'objectif de ce TP est de mettre en place votre espace de travail sur GitHub. Plutôt que de partir de zéro, nous allons "forker" (copier) le dépôt public de notre application fil rouge (Spring PetClinic) directement sur votre compte.

      Prérequis

      Disposer d'un compte GitHub avant de commencer

      Utiliser un compte GitHub existant si vous en avez déjà un

      Sinon, créer un compte sur https://github.com/signup

      Exercice 1 : Fork du dépôt

      Aller sur l'URL du dépôt fourni par le formateur :

      https://github.com/spring-projects/spring-petclinic

      En haut à droite de la page, cliquer sur le bouton "Fork"

      Laisser les paramètres par défaut (votre compte comme *Owner*) puis cliquer sur "Create fork"

      Félicitations, vous possédez désormais votre propre copie exacte du code source !

      Exercice 2 : Configuration de la clé SSH GitHub

      Avant de cloner en SSH, configurer une clé d'authentification sur votre poste puis dans GitHub.

      Générer une clé SSH

      ssh-keygen -t ed25519 -C "<VOTRE_EMAIL_GITHUB>"

      Quand ssh-keygen demande le chemin de stockage,

      laisser vide puis appuyer sur Entrée pour conserver le chemin par défaut (~/.ssh/id_ed25519)

      Quand ssh-keygen demande une passphrase,

      laisser vide puis appuyer sur Entrée (deux fois) pour ne pas avoir à la saisir à chaque push pendant le TP

      Ajouter la clé publique dans GitHub

      ouvrir le fichier ~/.ssh/id_ed25519.pub

      copier son contenu

      aller dans Settings > SSH and GPG keys > New SSH key

      coller la clé puis cliquer sur Add SSH key

      Tester la connexion SSH vers GitHub

      ssh -T git@github.com

      Exercice 3 : Configuration Git globale (une fois pour toutes)

      Configurer votre identité Git en global pour éviter de la ressaisir à chaque nouveau projet.

      git config --global user.name "<VOTRE_NOM>"
git config --global user.email "<VOTRE_EMAIL_GITHUB>"
git config --global init.defaultBranch main
git config --global --list

      Exercice 4 : Configuration de l'environnement local

      Pour éviter de modifier manuellement chaque commande dans les prochains TPs, nous allons définir votre identifiant GitHub dans un fichier de configuration.

      Créer un fichier gitops.env dans votre répertoire personnel et y ajouter votre identifiant (remplacer votre-identifiant par votre vrai nom d'utilisateur GitHub) :

      echo 'export GITHUB_USER="votre-identifiant"' > ~/gitops.env

      Charger cette variable dans votre terminal actuel :

      source ~/gitops.env

      Note : En cas de fermeture du terminal pendant la formation, relancer la commande source ~/gitops.env pour recharger votre identifiant

      Exercice 5 : Rapatriement local (Clone)

      Maintenant que votre usine existe sur votre compte cloud, nous allons la lier à votre machine locale.

      Sur la page de votre nouveau dépôt (vérifier que votre nom d'utilisateur est bien dans l'URL), cliquer sur le bouton vert Code puis copier l'URL HTTPS ou SSH

      Ouvrir un terminal sur votre machine et cloner le dépôt :

      git clone <URL_DE_VOTRE_NOUVEAU_DEPOT>
cd spring-petclinic

      Exercice 6 : Activation des GitHub Actions

      Par mesure de sécurité, GitHub bloque l'exécution automatique des pipelines de CI/CD sur les dépôts fraîchement forkés. Il faut les débloquer pour la suite du cours.

      Retourner sur la page GitHub de votre dépôt

      Cliquer sur l'onglet "Actions" (en haut, sous le nom du dépôt)

      Cliquer sur le bouton vert "I understand my workflows, go ahead and enable them" (Je comprends mes workflows, les activer)

      TP 4 : GitHub Actions

      > 💡 La minute GitOps : Pourquoi ne pas compiler sur notre machine ? > Traditionnellement, à ce stade, un développeur taperait docker build sur sa machine pour tester la création de l'image (l'ancienne méthode). > Mais rappelez-vous la philosophie de notre architecture : la machine du développeur n'est pas digne de confiance (versions différentes, cache local, etc.). Pour garantir l'immutabilité, c'est à notre usine logicielle (la CI) de compiler l'image. C'est exactement ce que nous allons configurer maintenant !

      Exercice 1: créer un premier workflow

      Switcher à la branche main du dépôt spring-petclinic

      git checkout main

      Créer le répertoire .github/workflows dans le dépôt spring-petclinic

      mkdir -p .github/workflows

      Créer le fichier .github/workflows/blank.yml

      il s'agit d'un fichier de configuration YAML qui décrit le workflow

      le nom du fichier n'est pas important, mais il doit avoir l'extension .yml ou .yaml

      name: Blank Workflow
on: [push, pull_request, workflow_dispatch]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4
      - name: Run a one-line script
        run: echo "Hello, world!"

      Pousser le fichier dans le dépôt distant

      git add .github/workflows/blank.yml
git commit -m "Ajout d'un premier workflow"
git push

      Lancer le Workflow

      le workflow peut être déclenché manuellement workflow_dispatch

      cliquer sur le bouton Run workflow

      observer la mise en attente puis l'exécution

      observer les traces des différentes actions

      en particulier, celles de l'action actions/checkout

      Modifier le fichier blank.yml pour ajouter deux jobs simples

      le premier dépend du job build mais pas le second

        test1:
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: test after build
        run: echo "after build"
  test2:
    runs-on: ubuntu-latest
    steps:
      - name: test in parallel with build
        run: echo "parallel with build"

      Pousser la modification

      git add .
git commit -m "Modifie le workflow"
git push

      Aller sur Github, dans la page Actions

      observer le workflow en cours d'exécution (déclenché par le push)

      noter que les jobs ajoutés sont pris en compte

      noter que le job test1 est en attente de la fin du job build

      cliquer sur les jobs pour vérifier leurs traces

      Exercice 2: création d'une action custom JavaScript

      Cette action GitHub Actions a pour but de construire une salutation personnalisée à partir de username et greeting, de l'afficher dans les logs, puis de la rendre disponible via l'output message pour les étapes suivantes du workflow.

      Résumé des actions à réaliser

      Créer une action GitHub locale JavaScript dans un dossier dédié

      Définir le contrat de l'action (inputs, exécution, fichier principal) dans action.yml

      Implémenter la logique de salutation dans index.js

      Déclarer les dépendances Node.js dans package.json puis installer avec npm install

      Documenter l'action dans README.md

      Versionner le projet et créer un workflow de test local

      Exécuter le workflow avec ACT pour valider le comportement

      Partie réflexion (à faire avant d'ouvrir la solution)

      Quels inputs sont vraiment nécessaires, et lesquels doivent avoir une valeur par défaut ?

      Comment exposer un output réutilisable dans un workflow appelant ?

      Quel message d'erreur afficher si un input obligatoire est absent ?

      Pourquoi tester localement avec ACT avant de pousser sur GitHub ?

      Corrigé guidé : action JavaScript locale qui lit username / greeting, affiche une salutation dans les logs, puis expose message en sortie

      Créer un répertoire à la racine du workspace de travail ~/my-custom-action-js/

      La structure du projet sera la suivante :

      my-custom-action-js/
├── action.yml          # Définition de l'action
├── index.js            # Code principal de l'action
├── package.json        # Gestion des dépendances Node.js
└── README.md           # Documentation

      Définir le fichier action.yml

      name: "Greet User Action"
description: "Une action qui salue un utilisateur avec un message personnalisé"
inputs:
  username:
    description: "Nom de l'utilisateur à saluer"
    required: true
  greeting:
    description: "Message de salutation"
    required: false
    default: "Bonjour"
outputs:
  message:
    description: "Le message généré par l'action"
runs:
  using: "node20"
  main: "index.js"

      Explications :

      inputs : Définit les entrées que l'utilisateur peut fournir dans le workflow

      runs.using : Indique l'environnement utilisé pour exécuter l'action (ici, Node.js 20)

      runs.main : Spécifie le fichier principal de l'action

      Ecrire le code de l'action dans index.js

      const core = require('@actions/core');

try {
    // Lire les entrées fournies par l'utilisateur
    const username = core.getInput('username');
    const greeting = core.getInput('greeting');

    // Construire le message
    const message = `${greeting}, ${username}!`;

    // Afficher le message dans les logs
    console.log(message);

    // Définir une sortie pour l'action
    core.setOutput("message", message);
} catch (error) {
    core.setFailed(`Une erreur est survenue : ${error.message}`);
}

      Définir les dépendances nécessaires dans package.json

      {
  "name": "greet-user-action",
  "version": "1.0.0",
  "description": "Une action GitHub qui salue un utilisateur",
  "main": "index.js",
  "scripts": {
    "lint": "eslint .",
    "test": "jest"
  },
  "dependencies": {
    "@actions/core": "^1.10.0"
  }
}

      Installer les dépendances

      npm install

      Ajouter la documentation de l'action

      # Greet User Action

Cette action salue un utilisateur avec un message personnalisé.

      Exercice 3: création d'une action custom Docker

      Cette action Docker locale a pour but de recevoir un message en entrée, de l'afficher dans les logs d'exécution, puis de publier output-message pour les étapes suivantes du workflow.

      Résumé des actions à réaliser

      Créer une action Docker locale dans un dossier dédié

      Définir le contrat de l'action (inputs, outputs, runs) dans action.yml

      Implémenter le comportement d'exécution dans entrypoint.sh

      Construire l'image d'action via Dockerfile

      Documenter l'action dans README.md

      Versionner le projet, créer un workflow de test et valider avec ACT

      Partie réflexion (à faire avant d'ouvrir la solution)

      Pourquoi encapsuler la logique dans un conteneur plutôt que dans du JavaScript pur ?

      Quelle différence entre afficher un message dans les logs et l'exposer comme output ?

      Quels risques si le script entrypoint.sh n'a pas les droits d'exécution ?

      Dans quels cas préférer une action Docker à une action composite ?

      Corrigé guidé : action Docker locale qui prend message, exécute un script d'entrée dans le conteneur, puis expose output-message

      Créer un répertoire à la racine du workspace de travail ~/my-custom-action-docker/

      La structure du projet sera la suivante :

      my-custom-action-docker/
├── Dockerfile          # Image Docker pour l'action
├── action.yml          # Définition de l'action
├── entrypoint.sh       # Script exécuté par l'action
└── README.md           # Documentation de l'action

      Définir l'action et ses paramètres

      name: "Docker Message Action"
description: "Une action Docker qui génère et affiche un message personnalisé"
inputs:
  message:
    description: "Le message à afficher"
    required: true
outputs:
  output-message:
    description: "Le message généré par l'action"
runs:
  using: "docker"
  image: "Dockerfile"
  args:

      Créer le fichier .github/workflows/push-image.yml

      ce fichier va contenir la configuration du workflow pour pousser l'image de l'application sur GitHub Container Registry

      le nom du fichier n'est pas important, mais il doit avoir l'extension .yml ou .yaml

      on réutilise ici aussi le cache Maven pour accélérer les builds

      Le workflow doit réaliser les actions suivantes dans cet ordre :

      se déclencher sur push, pull_request (branche main) et manuellement via workflow_dispatch

      demander les permissions minimales nécessaires (contents: read et packages: write)

      récupérer le code source avec actions/checkout

      extraire la version Maven depuis pom.xml et la stocker comme output (steps.get_version.outputs.version)

      installer Java 17 + activer le cache Maven (actions/setup-java avec cache: maven)

      normaliser le owner GitHub en minuscules pour construire une URL GHCR valide

      s'authentifier à ghcr.io avec le GITHUB_TOKEN

      construire l'image Docker via Spring Boot Buildpacks en la taguant avec la version Maven

      ajouter 3 tags à la même image (<version>, <sha>, latest)

      .github/workflows/push-image.yml
      name: Push Docker Image
on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]
  workflow_dispatch:

permissions:
  contents: read
  packages: write

jobs:
  push:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Extraire la version du pom.xml
        id: get_version
        run: |
          VERSION=$(mvn help:evaluate -Dexpression=project.version -q -DforceStdout)
          echo "version=$VERSION" >> $GITHUB_OUTPUT

      - name: Set up JDK 17
        uses: actions/setup-java@v4
        with:
          java-version: '17'
          distribution: 'temurin'
          cache: 'maven'
      - name: Set lower case owner name
        run: |
          echo "OWNER_LOWER=$(echo ${{ github.repository_owner }} | tr '[:upper:]' '[:lower:]')" >> $GITHUB_ENV

      - name: Log in to GitHub Container Registry
        uses: docker/login-action@v2
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build image with Spring Boot Buildpacks
        run: |
          ./mvnw -DskipTests \
            -Dspring-boot.build-image.imageName=spring-petclinic:${{ steps.get_version.outputs.version }} \
            spring-boot:build-image

      - name: Tag and push image to GHCR
        run: |
          docker tag spring-petclinic:${{ steps.get_version.outputs.version }} ghcr.io/${{ env.OWNER_LOWER }}/spring-petclinic:${{ steps.get_version.outputs.version }}
          docker tag spring-petclinic:${{ steps.get_version.outputs.version }} ghcr.io/${{ env.OWNER_LOWER }}/spring-petclinic:${{ github.sha }}
          docker tag spring-petclinic:${{ steps.get_version.outputs.version }} ghcr.io/${{ env.OWNER_LOWER }}/spring-petclinic:latest

          docker push ghcr.io/${{ env.OWNER_LOWER }}/spring-petclinic:${{ steps.get_version.outputs.version }}
          docker push ghcr.io/${{ env.OWNER_LOWER }}/spring-petclinic:${{ github.sha }}
          docker push ghcr.io/${{ env.OWNER_LOWER }}/spring-petclinic:latest

      Committer et pousser les changements

      Le workflow demande bien permissions: packages: write

      Mais le package spring-petclinic a été créé au niveau de votre compte utilisateur, et n'autorise pas automatiquement votre dépôt à écrire dessus

      Exercice 7 : ajout de probes

      Améliorer le manifest deployment.yaml existant en y ajoutant des probes Kubernetes pour assurer la haute disponibilité

      Les probes permettent à Kubernetes de vérifier l'état de santé de l'application et d'agir en conséquence (redémarrage automatique, routage du trafic)

      Trois types de probes seront ajoutées progressivement : startupProbe, livenessProbe, readinessProbe

      Étape 1 : Ajouter la probe de démarrage (startupProbe)

      La startupProbe indique à Kubernetes que l'application est en cours de démarrage et ne doit pas encore recevoir du trafic.

      Paramètres à configurer :

      startupProbe : vérifier que l'application a démarré correctement
      startupProbe:
  httpGet:
    path: /tp6/actuator/health/readiness   # Endpoint Spring Boot Actuator pour vérifier la disponibilité
    port: 8080                               # Port d'écoute du conteneur
  initialDelaySeconds: 10                   # Délai avant la première tentative (10s)
  periodSeconds: 10                         # Intervalle entre les tentatives (10s)
  timeoutSeconds: 2                         # Timeout pour chaque requête HTTP (2s)
  failureThreshold: 30                      # Nombre d'échecs tolérés avant conclusion d'échec (30 tentatives = 5 min)

      Explications des paramètres :

      initialDelaySeconds: 10 : laisse le temps au conteneur de démarrer sans être harcelé

      periodSeconds: 10 : teste toutes les 10 secondes pendant que l'app démarre

      failureThreshold: 30 : accepte jusqu'à 30 échecs consécutifs (environ 5 minutes) avant de déclarer le pod comme échoué

      Stratégie : permissive au démarrage (attendre que l'application soit vraiment prête)

      Étape 2 : Ajouter la probe de vie (livenessProbe)

      La livenessProbe indique à Kubernetes que l'application tourne toujours. Si elle échoue, le pod est redémarré.

      Paramètres à configurer :

      livenessProbe : détecter un blocage/deadlock d'application
      livenessProbe:
  httpGet:
    path: /tp6/actuator/health/liveness    # Endpoint dédié à la vérification de vie (vs. readiness)
    port: 8080
  initialDelaySeconds: 30                  # Attendre 30s après que le démarrage soit confirmé
  periodSeconds: 10                        # Vérifier toutes les 10 secondes
  timeoutSeconds: 2
  failureThreshold: 3                      # Redémarrer après 3 échecs consécutifs (~30 secondes)

      Explications des paramètres :

      initialDelaySeconds: 30 : plus élevé car l'application a déjà démarré (plus de tolérance après)

      failureThreshold: 3 : strict (redémarrage rapide en cas de problème)

      Stratégie : détection rapide des pannes (liveness = "est-ce que le conteneur est mort?")

      Étape 3 : Ajouter la probe de disponibilité (readinessProbe)

      La readinessProbe indique à Kubernetes si le pod peut recevoir du trafic. Les requêtes sont routées UNIQUEMENT vers les pods "ready".

      Paramètres à configurer :

      readinessProbe : contrôler quand le pod reçoit du trafic
      readinessProbe:
  httpGet:
    path: /tp6/actuator/health/readiness   # Même endpoint que startup
    port: 8080
  initialDelaySeconds: 20                  # Attendre 20s (moins strict que liveness)
  periodSeconds: 10
  timeoutSeconds: 2
  failureThreshold: 6                      # Retirer du trafic après 6 échecs (~60 secondes)

      Explications des paramètres :

      initialDelaySeconds: 20 : un peu moins que liveness (app démarrée mais peut prendre du temps à être "ready")

      failureThreshold: 6 : plus tolérant que liveness (on ne tue pas le pod, on le pause juste)

      Stratégie : contrôle du trafic (readiness = "peux-tu traiter des requêtes?")

      Résumé : les trois probes en contexte

      ProbeObjectifÉchec = ActionDécourage user?
      startupProbeAttendre le démarrageRedémarrage après 30 défauts (~5 min)Oui (app crée au lancement)
      livenessProbeDétecter deadlock/hangRedémarrage après 3 défauts (~30s)Non (redémarrage brutal)
      readinessProbeContrôler routage traficPause (~60s), puis redémarrageNon (révoqué temporairement)

      Exercice 3 : configuration ArgoCD

      Dans ArgoCD, créer une nouvelle application

      Créer petclinic-application.yaml

      Ce fichier décrit l'état désiré de l'application pour Argo CD

      Argo CD va :

      lire le chart Helm depuis GitHub

      rendre les templates Helm avec les paramètres fournis

      déployer le résultat dans le namespace cible

      apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: petclinic
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/<identifiant>/helm.git
    targetRevision: HEAD
    path: '.'
    helm:
      parameters:
      - name: "app.contextPath"
        value: "/tp9"
  destination:
    server: https://kubernetes.default.svc
    namespace: tp9-argocd
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true

      Lecture guidée du fichier :

      metadata.name: petclinic : nom de l'application visible dans Argo CD

      metadata.namespace: argocd : l'objet Application est stocké dans le namespace Argo CD

      spec.source.repoURL : dépôt Git qui contient le chart Helm

      spec.source.path: '.' : le chart est à la racine du dépôt

      spec.source.helm.parameters : surcharge Helm appliquée au rendu (ici app.contextPath=/tp9)

      spec.destination.namespace: tp9-argocd : namespace Kubernetes où l'application sera déployée

      syncPolicy.automated.prune: true : supprime les ressources devenues absentes du Git

      syncPolicy.automated.selfHeal: true : corrige automatiquement les écarts entre cluster et Git

      syncOptions: CreateNamespace=true : crée automatiquement le namespace cible s'il n'existe pas

      Ouvrir le fichier petclinic-application.yaml dans votre éditeur, puis remplacer manuellement https://github.com/<identifiant>/helm.git par https://github.com/<votre-identifiant-github>/helm.git. Sauvegarder le fichier.

      Exercice 8 : Orchestrer le déploiement avec les Resource Hooks (PreSync)

      Objectif : exécuter une tâche critique avant le déploiement applicatif (ex: migration de base de données)

      Principe : un hook PreSync démarre avant l'application des manifests applicatifs

      Important : la politique HookSucceeded supprime automatiquement le Job après succès pour éviter d'encombrer le cluster

      Créer le fichier db-migration-job.yaml dans le Chart Helm de l'application

      par exemple dans ~/kubernetes/petclinic/templates/

      apiVersion: batch/v1
kind: Job
metadata:
  name: db-migration-job
  annotations:
    argocd.argoproj.io/hook: PreSync
    argocd.argoproj.io/hook-delete-policy: HookSucceeded
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: migration
          image: busybox:1.36
          command: ["sh", "-c", "sleep 15; echo Migration OK"]

      Avec Helm, il n'y a rien à ajouter dans un fichier d'index : il suffit de créer ce YAML dans templates/, Helm l'inclura automatiquement au rendu

      Committer et pousser les changements

      cd ~/kubernetes/petclinic
git add templates/db-migration-job.yaml
git commit -m "Ajout d'un hook PreSync pour migration"
git push

      Dans ArgoCD, lancer une synchronisation et observer le comportement

      le Job PreSync démarre avec une icône d'ancre

      la phase Sync reste bloquée environ 15 secondes

      le Job disparaît automatiquement après succès

      l'application se met ensuite à jour

      Exercice 9 : Finalizers ArgoCD - supprimer ou protéger les ressources

      Objectif : démontrer que le finalizer ArgoCD contrôle le comportement à la suppression de l'Application.

      Avec finalizer : ArgoCD supprime aussi les ressources Kubernetes gérées

      Sans finalizer : l'Application ArgoCD est supprimée, mais les ressources restent en place (protection d'un environnement de production contre une suppression accidentelle)

      Vérifier l'état actuel de l'Application et des ressources

      kubectl get application petclinic -n argocd
kubectl get all,ing -n tp9-argocd

      Cas 1 : finalizer absent (ressources conservées)

      Vérifier que l'Application ne porte pas de finalizer

      kubectl get application petclinic -n argocd -o yaml | grep -A2 finalizers

      Supprimer l'Application

      kubectl delete application petclinic -n argocd
kubectl get all,ing -n tp9-argocd

      Constat attendu

      l'Application ArgoCD a disparu

      les Deployments, Services, Ingress et Pods sont toujours présents dans le namespace tp9-argocd

      Recréer l'Application pour le second cas

      kubectl apply -f petclinic-application.yaml -n argocd
kubectl get application petclinic -n argocd
kubectl get all,ing -n tp9-argocd

      Cas 2 : finalizer présent (suppression cascade)

      Ajouter explicitement le finalizer ArgoCD sur l'Application

      kubectl patch application petclinic -n argocd --type=merge -p '{"metadata":{"finalizers":["resources-finalizer.argocd.argoproj.io"]}}'

      Vérifier qu'il est bien présent

      kubectl get application petclinic -n argocd -o yaml | grep -A2 finalizers

      Supprimer à nouveau l'Application

      kubectl delete application petclinic -n argocd
kubectl get all,ing -n tp9-argocd

      Constat attendu

      l'Application est supprimée

      ArgoCD supprime aussi les ressources gérées dans tp9-argocd

      Message clé production

      garder le finalizer actif permet une suppression complète (pratique en environnement de test)

      retirer le finalizer protège les ressources applicatives d'une suppression accidentelle de l'objet Application (pratique pour des environnements sensibles)

      TP 10 : Le pont CI/CD - Automatiser la mise à jour du dépôt GitOps

      Aller dans Settings (de votre profil) > Developer settings > Personal access tokens > Tokens (classic)

      Cocher uniquement la case repo (Full control of private repositories)

      Aller dans les Settings de ce dépôt > Secrets and variables > Actions

      Cliquer sur New repository secret

      Secret : *Coller votre token*

      Exercice 2 : Ajouter le Job de mise à jour dans la CI

      Nous allons maintenant ajouter un dernier job à notre workflow du TP 4.

      Éditer le fichier .github/workflows/ci.yml dans votre dépôt applicatif

      Ajouter le job suivant à la suite de votre job de build/push :

        update-gitops:
    needs: build # Mettre ici le nom de votre job précédent (celui qui push sur GHCR)
    runs-on: ubuntu-latest
    steps:
      - name: Cloner le dépôt d'infrastructure
        uses: actions/checkout@v4
        with:
          repository: '<VOTRE_USER_GITHUB>/<VOTRE_DEPOT_INFRA>' # A REMPLACER (ex: <VOTRE_USER_GITHUB>/gitops-infra)
          token: ${{ secrets.GITOPS_PAT }}
          path: 'infra-repo' # On clone dans un sous-dossier pour ne pas écraser l'espace de travail

      - name: Mettre à jour la version de l'image
        working-directory: ./infra-repo/dev
        run: |
          kustomize edit set image spring-petclinic=ghcr.io/<VOTRE_USER_GITHUB>/spring-petclinic:${{ github.sha }}

      - name: Commit et Push vers le dépôt d'infrastructure
        working-directory: ./infra-repo
        run: |
          git config user.name "GitHub Actions Bot"
          git config user.email "actions@github.com"
          git add .
          git commit -m "Auto-update image to tag ${{ github.sha }}"
          git push

      (Attention : remplacer <VOTREUSERGITHUB> par votre vrai nom d'utilisateur GitHub et <VOTREDEPOTINFRA> par le nom de votre dépôt d'infrastructure)

      Exercice 3 : Configuration de l'authentification (GHCR)

      L'Image Updater a besoin d'un jeton pour lister les tags disponibles sur votre registre privé.

      1. Créer le secret : Utiliser votre GitHub Token (scope read:packages).

      kubectl create secret docker-registry ghcr-pull-secret \
  -n argocd \
  --docker-server=ghcr.io \
  --docker-username=$GITHUB_USER \
  --docker-password=<VOTRE_TOKEN>

      2. Configurer le registre : Créer une ConfigMap nommée argocd-image-updater-config.yaml.

      apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-image-updater-config
  namespace: argocd
data:
  registries.conf: |
    registries:
      - name: GitHub Container Registry
        prefix: ghcr.io
        api_url: https://ghcr.io
        credentials: pullsecret:argocd/ghcr-pull-secret

      3. Appliquer et redémarrer le contrôleur pour charger la configuration :

      kubectl apply -f argocd-image-updater-config.yaml
kubectl rollout restart deployment argocd-image-updater-controller -n argocd

      Exercice 4 : Configuration de l'Application Argo CD

      Nous allons piloter l'automate via des annotations. L'alias choisi ici est spring-petclinic.

      1. Ouvrir votre petclinic-application.yaml et remplacer manuellement ghcr.io/<USER>/spring-petclinic par ghcr.io/<votre-identifiant-github>/spring-petclinic, puis https://github.com/<USER>/helm.git par https://github.com/<votre-identifiant-github>/helm.git. Sauvegarder le fichier.

      apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: petclinic
  namespace: argocd
  annotations:
    argocd-image-updater.argoproj.io/image-list: spring-petclinic=ghcr.io/<USER>/spring-petclinic
    argocd-image-updater.argoproj.io/spring-petclinic.update-strategy: semver
    argocd-image-updater.argoproj.io/spring-petclinic.allow-tags: regexp:^([0-9]+)\.([0-9]+)\.([0-9]+)$
    argocd-image-updater.argoproj.io/spring-petclinic.helm.image-tag: image.tag
    argocd-image-updater.argoproj.io/write-back-method: git
spec:
  project: default
  source:
    repoURL: https://github.com/<USER>/helm.git
    targetRevision: HEAD
    path: '.'
    helm:
      parameters:
      - name: "image.tag"
        value: "4.0.3" # Version de départ
      - name: "app.contextPath"
        value: "/tp11"
  destination:
    server: https://kubernetes.default.svc
    namespace: default
  syncPolicy:
    automated:
      prune: true
      selfHeal: true

      Note : Grâce à la méthode "git", l'Image Updater va directement pousser un commit dans notre dépôt pour mettre à jour la version de l'image, respectant ainsi à 100% le principe du GitOps !

      Exercice 2 : Automatiser avec les ApplicationSets

      Le pattern App of Apps est puissant, mais nécessite de créer un fichier YAML par application enfant. Nous allons maintenant utiliser un ApplicationSet pour générer ces applications dynamiquement à partir d'une arborescence Git.

      Dans votre dépôt d'infrastructure actuel, créer la structure de fichier suivante à la racine

      le dossier apps doit être créé juste à côté du dossier cluster-config

      apps/
├── dev/
│   ├── deployment.yaml
│   └── service.yaml
├── staging/
│   ├── deployment.yaml
│   └── service.yaml
└── prod/
    ├── deployment.yaml
    └── service.yaml

      deployment.yaml

      apiVersion: apps/v1
kind: Deployment
metadata:
  name: guestbook
spec:
  replicas: 2
  selector:
    matchLabels:
      app: guestbook
  template:
    metadata:
      labels:
        app: guestbook
    spec:
      containers:
        - name: guestbook
          image: nginx
          ports:
            - containerPort: 80

      service.yaml

      apiVersion: v1
kind: Service
metadata:
  name: guestbook
spec:
  type: ClusterIP
  selector:
    app: guestbook
  ports:
    - port: 80
      targetPort: 80

      Créer une ApplicationSet

      guestbook-applicationset.yaml

      apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: guestbook-environments
  namespace: argocd
spec:
  generators:
    - git:
        repoURL: https://github.com/<identifiant>/<votre-repo-gitops>.git
        revision: HEAD
        directories:
          - path: apps/*
  template:
    metadata:
      name: '{{path.basename}}-guestbook'
    spec:
      project: default
      source:
        repoURL: https://github.com/<identifiant>/<votre-repo-gitops>.git
        targetRevision: HEAD
        path: '{{path}}'
      destination:
        server: https://kubernetes.default.svc
        namespace: '{{path.basename}}'
      syncPolicy:
        automated:
          prune: true
          selfHeal: true
        syncOptions:
          - CreateNamespace=true

      Appliquer le fichier guestbook-applicationset.yaml

      kubectl apply -f guestbook-applicationset.yaml -n argocd

      constater dans ArgoCD que 3 nouvelles applications ont été créées

        Créer une nouvelle branche pour votre modification

        git checkout -b my-feature

        vérifier que vous êtes bien sur la branche my-feature

        git branch

        ajouter une ligne de texte à la fin du fichier

        Enregistrer la modification

        git add README.md
git commit -m "Ajout d'une ligne dans le README"

        Pousser la branche vers le dépôt distant

        git push -u origin my-feature

        vérifier que la branche est bien poussée vers le dépôt distant

        git branch -r

        Exercice 2 : Créer une Pull Request

        Vérifier que la branche main est sélectionnée comme branche de base

        Cliquer sur le bouton Create pull request

        Ajouter un titre et une description pour la Pull Request

        par exemple, Ajout d'une ligne dans le README

        Cliquer sur le bouton Create pull request

        Observer que la Pull Request est créée et que les modifications sont visibles
